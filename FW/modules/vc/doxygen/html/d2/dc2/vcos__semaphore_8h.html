Detailed Description

Semaphores

This provides counting semaphores. Semaphores are not re-entrant. On sensible operating systems a semaphore can always be posted but can only be taken in thread (not interrupt) context. Under Nucleus, a LISR cannot post a semaphore, although it would not be hard to lift this restriction.

Timeout

On both Nucleus and ThreadX a semaphore can be taken with a timeout. This is not supported by VCOS because it makes the non-timeout code considerably more complicated (and hence slower). In the unlikely event that you need a timeout with a semaphore, and you cannot simply redesign your code to avoid it, use an event flag (vcos_event_flags.h).

Changes from Nucleus:

Semaphores are always "FIFO" - i.e. sleeping threads are woken in FIFO order. That's because:

  • there's no support for NU_PRIORITY in threadx (though it can be emulated, slowly)
  • we don't appear to actually consciously use it - for example, Dispmanx uses it, but all threads waiting are the same priority.

VCOS_INLINE_DECL VCOS_STATUS_T vcos_semaphore_wait ( VCOS_SEMAPHORE_T *  sem)

Wait on a semaphore.

There is no timeout option on a semaphore, as adding this will slow down implementations on some platforms. If you need that kind of behaviour, use an event group.

On most platforms this always returns VCOS_SUCCESS, and so would ideally be a void function, however some platforms allow a wait to be interrupted so it remains non-void.

Parameters
semSemaphore to wait on
Returns
VCOS_SUCCESS - semaphore was taken. VCOS_EAGAIN - could not take semaphore
